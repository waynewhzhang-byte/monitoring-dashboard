# 手工同步架构设计

## 当前架构问题

### 1. 设备同步
- **当前**：自动定期从OpManager LISTDEVICE API获取所有设备并同步
- **问题**：每次动态获取，可能导致数据不一致，无法控制同步时机

### 2. 接口同步
- **当前**：自动定期从OpManager获取所有监控设备的接口
- **问题**：直接从OpManager获取设备列表，而不是基于数据库中已同步的设备

## 优化后的架构

### 1. 设备同步流程（手工触发）

```
用户点击"同步设备"按钮
    ↓
调用 /api/devices/sync (POST)
    ↓
从OpManager LISTDEVICE API获取设备列表
    ↓
遍历设备，执行 upsert 到 Device 表
    ↓
返回同步结果（成功/失败数量）
```

### 2. 接口同步流程（基于数据库设备）

```
用户点击"同步接口"按钮
    ↓
调用 /api/interfaces/sync (POST)
    ↓
从数据库 Device 表获取所有 isMonitored=true 的设备
    ↓
遍历每个设备：
  - 使用 device.opmanagerId 或 device.name 作为参数
  - 调用 OpManager getInterfaces API
  - 同步接口到 Interface 表
    ↓
返回同步结果（设备数、接口数）
```

### 3. 数据一致性保证

- **设备表是数据源**：接口同步基于Device表中的设备，而不是直接从OpManager获取
- **手工同步**：所有同步操作由管理员手工触发，确保数据一致性
- **幂等性**：使用 upsert 操作，多次同步不会产生重复数据

## 实施计划

### 阶段1：设备同步API
- 创建 `/api/devices/sync` 端点
- 从OpManager获取设备并同步到数据库
- 返回同步统计信息

### 阶段2：接口同步API优化
- 修改 `/api/interfaces/sync` 
- 改为从数据库Device表获取设备列表
- 遍历设备并同步接口

### 阶段3：UI更新
- 在设备管理页面添加"同步设备"按钮
- 更新接口管理页面的"同步接口"按钮说明
- 显示同步进度和结果

### 阶段4：移除自动同步（可选）
- 可以保留自动同步作为后台任务
- 或者完全移除，只保留手工同步
